# 交互编排 (Interactive Orchestration)

## 1. 组件通信与状态同步 (Component Communication)

组件之间不应直接修改对方的 DOM 或私有状态，必须通过 **事件驱动 (Event-driven)** 或 **状态分发 (State Dispatching)** 进行同步。

- **单向数据流**: 父级通过 Props 控制子级，子级通过 Emit 事件通知父级。
- **兄弟组件通信**: 必须通过共享的状态仓库（Store）或 Context 进行信号中转。
- **示例**: 
  - *输入信号*: 用户在 `SearchInput` 输入关键词。
  - *协同行为*: `Sidebar` 过滤菜单，`ContentStage` 高亮匹配项。
  - *实现要求*: `SearchInput` 发送 `onSearchChange` 事件，中央 Store 更新状态，下游组件感知状态变更。

---

## 2. 加载编排 (Loading Orchestration)

消除“闪烁感”和“布局抖动”，通过多层级的加载态确保交互的平滑过渡。

### 2.1 骨架屏 (Skeleton - Macro)
- **触发条件**: 首次数据获取，预计时间 > 500ms。
- **规范**: 骨架屏的轮廓应与最终渲染的组件 **高度一致**，避免数据填充时的页面跳动。

### 2.2 加载指示器 (Spinner - Micro)
- **触发条件**: 局部更新（如：点击“保存”）、按钮异步执行。
- **规范**: Spinner 应内嵌于触发源内部（如按钮内文字左侧），不应遮挡全局视图。

### 2.3 渐进式披露 (Progressive Disclosure)
- **逻辑**: 先显示已缓存的数据，同时显示微小的 Loading 指示器，数据刷新后再更新视图。

---

## 3. 反馈优先级 (Feedback Hierarchy)

根据交互的重要程度和破坏性，选择合适的反馈容器。

| 级别 | 容器 (Container) | 场景 | 交互约束 |
| :--- | :--- | :--- | :--- |
| **P0** | **Modal** | 危险操作确认、核心任务创建。 | 阻塞操作，必须交互后关闭。 |
| **P1** | **Inline Alert** | 表单校验错误、局部权限提示。 | 紧随触发源，长期存在直到错误消除。 |
| **P2** | **Toast (Message)** | 操作成功反馈、低风险后台提醒。 | 自动消失，不干扰当前任务流。 |
| **P3** | **Tooltip** | 按钮功能说明、复杂术语解释。 | Hover/Focus 触发，瞬态。 |

---

## 4. 焦点管理 (Focus Management)

确保全键盘操作的连续性与逻辑性，防止用户在复杂 UI 中“迷失光标”。

### 4.1 焦点陷阱 (Focus Trap)
- **模态框 (Modal)**: 激活时，焦点必须被限制在 Modal 内部，按 `Tab` 键应在 Modal 元素间循环，按 `Esc` 应关闭并返回原焦点。

### 4.2 自动聚焦与恢复
- **进入**: 进入新页面或打开抽屉时，首个主操作按钮（CTA）或首个输入框应自动获得焦点。
- **退出**: 关闭悬浮层后，焦点必须 **精准返回** 到原本触发该层的元素上。

---

## 5. 事件处理规范 (Event Handling)

在高频交互场景下，必须实施策略性流量控制以保证流畅度。

### 5.1 防抖 (Debouncing)
- **适用**: 远程搜索输入、窗口 `Resize`。
- **标准**: 默认等待时间为 **300ms**。

### 5.2 节流 (Throttling)
- **适用**: 拖拽操作、页面 `Scroll`、鼠标轨迹跟踪。
- **标准**: 默认采样率不低于 **60fps (16ms)** 或固定每 **100ms** 执行一次。

### 5.3 预测性拦截
- 在发起写操作（POST/DELETE）时，立即拦截按钮点击事件并设为 `loading/disabled`，防止短时间内重复提交（Double Tap Protection）。

---
*交互编排的最终目标是构建“确定性”：用户每一次点击，都能得到预期的、有节奏的反馈。*
