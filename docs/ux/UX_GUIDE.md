# Codex 统一交互指南 (Unified UX/Interaction Guide)

本指南定义 Codex UX 的两层规则:

- **外部基线**: 来自 WCAG 2.2、WAI-ARIA、NIST AI RMF 等规范或研究。
- **内部约定**: Codex 针对编码协作场景做出的产品决策。

外部来源总表见: [**UX 权威来源基线**](./UX_AUTHORITY_BASELINE.md)

## 1. 核心设计哲学 (Design Philosophy)

Codex 的 UI 不是普通网页，而是一个**高性能协作工作台**。

- **低认知负荷**: 用户不应频繁思考“我在哪”“下一步是什么”。
- **上下文优先**: 当前任务、当前文件、当前审批对象优先于全局装饰。
- **即时可感**: 系统必须尽早给出稳定反馈，避免让用户怀疑操作是否生效。
- **可检查与可恢复**: 尤其是 AI 参与的流程，必须允许查看依据、审批、撤销和回退。

## 2. 交互策略 (Interaction Strategy)

### 2.1 预见性交互 (Predictive UX)

AI 可以帮助用户减少机械操作，但不得越过透明度与控制权边界。

- **上下文建议**: 可以基于当前文件、错误信息或会话历史提供下一步建议。
- **预加载与预取**: 可以提前准备相关面板或数据，但不能伪造“已经执行”。

### 2.2 风险分层 (Risk-tiered Interaction)

高风险动作应提高可见性与审查强度，但不要求一刀切使用相同摩擦形式。

- **低风险**: 直接执行，必要时提供撤销。
- **中风险**: 展示范围、差异和撤销路径。
- **高风险**: 执行前要求显式确认、审批或更强校验。

## 3. 核心空间隐喻: Rail-Stage 模式

整个系统保持稳定的空间逻辑。

| 区域 | 角色 | 交互行为 |
| :--- | :--- | :--- |
| **App Rail** | 导航与全局切换 | 常驻，承载一级入口。 |
| **Side Panel** | 配置、二级目录、上下文工具 | 辅助当前任务，可折叠。 |
| **Stage** | 线程、编辑器、diff、结果主体 | 最高视觉权重，承载主要工作流。 |
| **Mode Strip** | 页面标题、状态、全局动作 | 承载当前上下文与主要 CTA。 |

## 4. 交互金律 (Golden Rules)

### 4.1 可预测性 (Predictability)

- **位置一致**: 同一流程里的主要动作、危险动作和辅助动作要保持稳定位置与视觉关系。
- **语义清楚**: 按钮、菜单、快捷键和状态名必须直接说明结果。
- **可交互项应可感知**: 指针、焦点、hover 或按压态至少要有一种清楚反馈。

### 4.2 反馈及时 (Timely Feedback)

- **立即确认已收到操作**: 点击、快捷键、提交后立刻反馈。
- **长任务逐步披露进度**: 使用局部 loading、队列状态、流式输出或阶段提示，而不是静默等待。

### 4.3 容错与恢复 (Recovery)

- **撤销优先**: 能安全撤销的流程，优先提供撤销而不是过度确认。
- **不可逆操作才升级防护**: 真正不可逆或高影响动作再使用模态确认、审批或重新认证。
- **AI 动作可检查**: 涉及 AI 的修改建议必须能看 diff、看来源、看范围。

## 5. 通用交互模式 (Interaction Patterns)

### 列表与卡片 (Lists and Cards)

- **密度可控**: 默认只展示决策所需信息，次要操作按需露出。
- **整卡可点击时要避免事件冲突**: 卡片主区域与内部按钮必须有清晰事件边界。

### 模态框与侧滑窗 (Modals vs. Drawers)

- **模态框**: 用于中断式、必须完成或必须确认的任务。
- **侧滑窗**: 用于补充信息、上下文配置和不必中断主流程的编辑。

## 6. 无障碍与标准 (Accessibility and Standards)

- **WCAG 2.2 AA** 是当前合规基线。
- **全键盘可操作** 是默认要求，不是可选优化。
- **焦点、状态消息、对比度和减弱动效** 必须在所有主要工作流中被验证。
