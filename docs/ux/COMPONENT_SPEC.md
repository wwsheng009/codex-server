# 组件设计规范 (Component Specification)

## 1. 核心设计价值观 (Core Design Values)
遵循 Ant Design 5.0 的设计哲学，Codex 的所有组件必须体现以下四项核心价值：

| 价值观 | 描述 | 设计体现 |
| :--- | :--- | :--- |
| **Natural (自然)** | 符合直觉，减少认知负担。 | 采用物理世界的动效（Spring Physics），间距符合视觉比例。 |
| **Certain (确定)** | 提供可预测的反馈，消除歧义。 | 统一的状态响应（Hover/Active），清晰的错误提示定位。 |
| **Meaningful (有意义)** | 每一个视觉元素都有其功能目的。 | 拒绝装饰性 UI，高亮仅用于引导焦点或表达状态。 |
| **Restrained (克制)** | 避免过度设计，保持界面简洁。 | 减少边框使用，利用留白划分区域，默认隐藏低频操作。 |

## 2. 表单设计模式 (Form Patterns)

### 布局与對齐
- **顶左对齐 (Top-aligned Labels)**: 默认采用标签置于输入框上方的布局，以提供最佳的垂直阅读流和扩展性。
- **必填项标识**: 星号 (`*`) 置于标签左侧，使用 `--danger` 色值。

### 校验逻辑
- **行内校验 (Inline Validation)**: 校验应在 `onBlur` 或用户停止输入 500ms 后触发，而非实时打断。
- **反馈等级**: 
  - `Error`: 红色描边 + 底部文字描述。
  - `Warning`: 橙色描边，不阻塞提交。
  - `Validating`: 渐变呼吸态或 Loading 图标。

### 联动逻辑 (Dependency-based Logic)
- **动态显隐**: 基于父级字段的值动态渲染子字段。
- **状态继承**: 禁用父容器时，其内部所有表单项应同步进入 `disabled` 状态。

## 3. 反馈与通知模式 (Feedback Patterns)

### 容器选择标准
| 模式 | 使用场景 | 交互特性 |
| :--- | :--- | :--- |
| **Modal (对话框)** | 中断性强、聚焦单一任务（如确认删除、快速创建）。 | 居中显示，强制独占。 |
| **Drawer (抽屉)** | 承载复杂表单、详情预览、配置项。 | 从右侧滑出，支持保持上下文。 |
| **Popover (气泡卡片)** | 简单设置、快捷操作、补充说明。 | 邻近触发源，轻量级。 |

### 通知规范
- **Hook-based Notifications**: 全局通知应通过 `useNotification` 或 `useMessage` 调用，确保状态隔离。
- **自动关闭**: 成功信息 (Success) 3s 后自动消失；警告 (Warning) 与错误 (Error) 需手动关闭。

### 占位与结果
- **Skeleton (骨架屏)**: 在初次加载超过 1s 时显示，模拟最终布局的灰度色块。
- **Result (结果页)**: 任务完成（Success）、权限不足（403）或网络错误（500）时，使用标准化的 Result 组件引导用户。

## 4. 导航模式 (Navigation Patterns)

### 层级导航
- **Breadcrumbs (面包屑)**: 超过三级深度的页面必须提供，允许跨级跳转。
- **Menu Grouping**: 侧边栏菜单需按逻辑分组，并使用 `Collapsible` 保持空间整洁。

### 引导与教育
- **Tour (漫游引导)**: 针对新功能或初次登录用户，实施原子化的步骤引导。
- **Empty State**: 当列表或容器无数据时，不仅要显示“暂无数据”，还应提供“去创建”或“导入”的行动按钮。
